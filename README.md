# Search Local Files

## Overview

此 Skill 用于处理用户搜索本地文件、以及将文件上传至云端的需求，支持 PC 端和手机端。**两端工具名称不同，参数也有差异**，须严格按设备类型选择对应工具调用。禁止通过其他方式（如 system.run、exec、shell 命令）操作本地文件。

完整流程分为两个阶段：
1. **搜索阶段**：调用 `SearchLocalFiles`（PC）或 `search_local_files`（手机）搜索文件，返回文件列表
2. **上传阶段**：仅当用户明确表达"发给我"、"下载"、"我要用这个文件"等意图时触发，调用 `UploadLocalFiles`（PC）或 `upload_local_files`（手机）将文件上传至云端，返回下载链接

---

## 工具调用方式

本 Skill 运行在 tianxiclaw 平台，MCP server 已注册在 mcp manager 中。
通过 as-node 链路调用工具，步骤如下：

1. 通过 `as_node_get_nodes` 获取所有可用节点（云端场景 node_id 传空）
2. 根据节点列表确定目标设备（见 Step 1：设备选择）
3. 通过 `as_node_get_mcp_list` 找到目标节点下的 MCP server
4. 通过 `as_node_get_tool_list` 找到对应工具
5. 通过 `as_node_call_tool` 执行该工具

**关键约束：**
- **PC 端搜索使用 `SearchLocalFiles`，手机端搜索使用 `search_local_files`**，两者参数不同，不可混用
- **PC 端上传使用 `UploadLocalFiles`，手机端上传使用 `upload_local_files`**，不可混用
- **禁止使用** `system.run`、`exec`、`shell` 命令操作本地文件
- 操作目标是**用户设备上的文件**，不是 OpenClaw workspace 中的文件
- **暂不支持**按 C 盘/D 盘等具体盘符路径搜索（仅 PC 端限制）

## MCP Tool 参数说明

### PC 端搜索：SearchLocalFiles

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `query` | string | 是 | 用户查询原文 |
| `key_word` | array | 否 | 关键词数组，提取核心名词 |
| `time_range` | array | 否 | 时间范围数组，格式: `[{"start_time": "2025-03-23", "end_time": "2025-03-23"}]` |
| `location` | array | 否 | 地点数组 |
| `type` | array | 否 | 文件类型: `all`/`image`/`video`/`audio`/`doc`/`zip`/`folder` |
| `suffix` | array | 否 | 文件后缀数组 |
| `device` | array | 否 | 预留字段，默认空 |

---

### 手机端搜索：search_local_files

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `query` | string | 是 | 用户输入的完整查询语句，用于向量语义搜索 |
| `key_word` | array | 否 | 关键词列表，多词间为 OR 关系；单个字符串内空格分隔表示 AND（如 `["火锅 啤酒", "烤鱼"]` 表示 `(火锅 AND 啤酒) OR 烤鱼`）。**仅含核心语义内容**（事件、场景、对象、人物、项目名等），⚠️ 严格排除：时间词（昨天/国庆）、类型词（照片/视频）、设备词（手机/电脑） |
| `time_range` | array | 否 | 时间区间数组，多个区间为 OR 关系，格式: `[{"start_time": "YYYY-MM-DD", "end_time": "YYYY-MM-DD"}]` |
| `location` | array | 否 | 地点数组，OR 关系，支持国家/省/市/区县/4A及以上景区或著名地标 |
| `type` | array | 否 | 文件类型: `all`/`image`/`video`/`audio`/`doc`/`zip`（⚠️ 不支持 `folder`；`all` 不能与其他类型同时出现） |
| `suffix` | array | 否 | 文件后缀名列表，OR 关系，置空表示不限后缀 |
| `device` | array | 否 | 预留字段，默认空 |

---

### 上传：UploadLocalFiles（PC 端）/ upload_local_files（手机端）

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `data` | array | 是 | 待上传文件列表，每项包含 `path`（必填）和 `uri`（可选） |
| `data[].path` | string | 是 | 文件的本地绝对路径 |
| `data[].uri` | string | 否 | content URI，有值时优先用于读取文件内容，无需额外存储权限 |

**上传限制规则（上传前须按此规则过滤文件列表）：**

1. **单文件大小限制**：排除所有 >500MB 的文件，被排除的文件返回错误码
2. **排序**：剩余文件按**修改时间**排序后依次入队
3. **总大小限制**：累计总大小超过 1GB 时，超出的那个文件及后续文件不上传，返回这些文件的名单及错误码
4. **文件数量限制**：文件个数达到 20 个后，第 21 个及后续文件不上传，返回这些文件的名单及错误码

> 注：大小限制和数量限制同时生效，以先触发者为准。

## Workflow

### Step 1: 设备选择

调用 `as_node_get_nodes` 获取所有可用节点，按以下逻辑处理：

**情况 A：用户已在请求中明确指定设备**
- 用户说"电脑上"、"PC 上"、"电脑里" → 直接定位到 PC 节点，搜索使用 `SearchLocalFiles`
- 用户说"手机上"、"手机里" → 直接定位到手机节点，搜索使用 `search_local_files`
- 若指定设备不在节点列表中，回复："未检测到你的 [PC/手机]，请确认设备已连接并开启相关权限。"

**情况 B：用户未指定设备，只有 1 个节点**
- 直接使用该节点，根据节点类型选择对应工具，无需询问

**情况 C：用户未指定设备，有多个节点**
- 向用户展示设备列表，让用户选择：
  > "检测到以下设备，你想搜索哪个设备上的文件？
  > 1. [设备名称/类型 1]
  > 2. [设备名称/类型 2]
  > ..."
- 等待用户选择后继续执行

**情况 D：没有任何节点**
- 回复："未检测到任何可用设备，请确认设备已连接并开启相关权限。"

---

### Step 2: 处理模糊查询

如果用户只说"找文件"、"搜一下"等模糊表达（没有提供任何关键词、类型、时间），**必须追问**：
> "请告诉我你要找什么类型的文件？比如照片、视频、文档，或者文件名关键词。"

---

### Step 3: 检查盘符搜索（仅 PC 端）

如果目标设备是 PC，且用户要求按 C 盘/D 盘/E 盘等具体盘符搜索：
> "暂不支持按 C 盘/D 盘等具体盘符路径搜索，我帮你用其他条件搜索。"

移除盘符限制，用剩余条件（关键词、时间、类型）继续搜索。手机端跳过此步骤。

---

### Step 4: 解析查询参数

#### 时间解析（使用用户本地时区）

| 用户表述 | 转换结果 |
|---------|---------|
| 今天 | 当天日期 |
| 昨天 | 前一天日期 |
| 前天 | 前两天日期 |
| 上周 | 上周一至上周日 |
| 上周X | 上周具体星期几 |
| 上个月 | 上月1日至月末 |

#### 文件类型识别

| 用户表述 | type 参数 | PC 端 | 手机端 |
|---------|----------|:-----:|:------:|
| 照片/图片/截图 | `image` | ✅ | ✅ |
| 视频/录像/电影 | `video` | ✅ | ✅ |
| 音乐/歌曲/音频 | `audio` | ✅ | ✅ |
| 文档/Word/Excel/PPT/PDF | `doc` | ✅ | ✅ |
| 压缩包/zip/rar | `zip` | ✅ | ✅ |
| 文件夹/目录 | `folder` | ✅ | ❌ |

> 手机端用户若询问"文件夹"，回复："手机端暂不支持按文件夹搜索，我帮你用关键词搜索相关文件。"然后移除 `folder` 类型，用其他条件继续搜索。

#### 关键词提取

| 示例查询 | key_word | 说明 |
|---------|----------|------|
| "火锅的照片" | `["火锅"]` | "照片"为类型词，不入 key_word |
| "工作文档" | `["工作"]` | "文档"为类型词，不入 key_word |
| "成都旅游视频" | `["旅游"]` | "成都"入 location，"视频"为类型词，均不入 key_word |
| "昨天的截图" | `[]` | "昨天"入 time_range，"截图"入 type，key_word 为空 |
| "火锅 啤酒的视频"（手机端） | `["火锅 啤酒"]` | AND 语义用空格分隔写在同一字符串内 |

#### 地点识别

识别地理位置名词（国家/省/市/区县/知名景区地标），提取到 `location` 参数，并从 `key_word` 中移除。

---

### Step 5: 调用搜索工具

在已确定的目标节点下，找到对应 MCP server，按设备类型调用：
- PC 端 → `SearchLocalFiles`
- 手机端 → `search_local_files`

---

### Step 6: 处理搜索结果

按 MCP 返回格式原样将文件列表返回给用户：
```json
{
  "files": [...],
  "lightWords": [...],
  "code": 0,
  "description": ""
}
```

**边界情况处理：**
- `files` 为空：回复"未找到匹配的文件，建议换个关键词或扩大时间范围再试试"
- `code != 0`：回复"搜索服务暂时不可用，请稍后重试"

搜索完成后，**不自动触发上传**，等待用户进一步操作。

---

### Step 7: 上传文件（按需触发）

**仅当用户明确表达以下意图时触发**，如：
- "发给我"、"发过来"
- "我要下载"、"给我下载链接"
- "我要用这个文件"
- 或其他明确需要获取文件内容的表达

#### 7.1 确定上传文件列表

- 若用户指定了特定文件（如"把第一个发给我"），仅上传指定文件
- 若用户未指定（如"都发给我"），上传搜索结果中的全部文件

#### 7.2 过滤与排序

按以下规则处理待上传列表（**上传前必须执行**）：

1. 排除所有单文件 >500MB 的文件，并告知用户哪些文件因超大被排除
2. 剩余文件按**修改时间**排序
3. 依次累加文件大小，累计超过 1GB 时，停止添加后续文件
4. 若文件数量达到 20 个，停止添加后续文件
5. 若有文件因第 3、4 步未被上传，告知用户哪些文件未能上传及原因

#### 7.3 构造上传参数

从搜索结果的每个文件中取 `path`，若存在 `uri` 字段则一并传入：

```json
{
  "data": [
    { "path": "/storage/emulated/0/DCIM/火锅.jpg", "uri": "content://media/..." },
    { "path": "/storage/emulated/0/DCIM/成都.jpg" }
  ]
}
```

#### 7.4 调用上传工具

在同一设备节点下，按设备类型调用对应上传工具，传入过滤后的文件列表：
- PC 端 → `UploadLocalFiles`
- 手机端 → `upload_local_files`

#### 7.5 处理上传结果

- 上传成功：将每个文件的下载链接返回给用户
- 部分失败：成功的返回链接，失败的告知文件名及失败原因
- 全部失败：回复"文件上传失败，请稍后重试"

## Examples

### Example 1: 搜索后用户要求发送文件
用户："帮我找手机里昨天去成都吃火锅的照片，找到了发给我"

**搜索阶段**，调用 `search_local_files`：
```json
{
  "query": "帮我找昨天去成都吃火锅的照片",
  "key_word": ["火锅"],
  "time_range": [{"start_time": "2025-03-23", "end_time": "2025-03-23"}],
  "location": ["成都"],
  "type": ["image"],
  "suffix": [],
  "device": []
}
```

搜索返回 3 张照片后，用户意图中已包含"发给我"，直接进入上传阶段。

**上传阶段**，过滤检查（3 个文件均 <500MB，总大小 <1GB，数量 <20），调用 `upload_local_files`（手机端）：
```json
{
  "data": [
    { "path": "/storage/emulated/0/DCIM/IMG_001.jpg", "uri": "content://media/001" },
    { "path": "/storage/emulated/0/DCIM/IMG_002.jpg", "uri": "content://media/002" },
    { "path": "/storage/emulated/0/DCIM/IMG_003.jpg", "uri": "content://media/003" }
  ]
}
```

返回 3 个下载链接给用户。

---

### Example 2: 搜索后用户再次触发上传
用户："帮我找电脑上的工作文档"

**搜索阶段**，调用 `SearchLocalFiles`，返回文件列表后等待用户操作。

用户："把第二个发给我"

**上传阶段**，仅上传第二个文件，调用 `UploadLocalFiles`（PC 端）：
```json
{
  "data": [
    { "path": "C:/Users/xxx/Documents/工作计划.docx" }
  ]
}
```

返回下载链接给用户。

---

### Example 3: 上传遇到大文件限制
搜索结果包含 4 个文件，其中 1 个为 600MB 的视频。

告知用户："以下文件超过 500MB 限制，无法上传：大视频.mp4（600MB）"

剩余 3 个文件正常上传，返回下载链接。

---

### Example 4: 多设备 → 让用户选择
用户："帮我找昨天拍的视频"

`as_node_get_nodes` 返回 PC + 手机两个节点：

回复：
> "检测到以下设备，你想搜索哪个设备上的文件？
> 1. 我的电脑（PC）
> 2. 我的手机"

用户选择手机后，调用 `search_local_files` 继续执行。

---

### Example 5: 模糊查询 → 追问
用户："找文件"

回复："请告诉我你要找什么类型的文件？比如照片、视频、文档，或者文件名关键词。"

---

### Example 6: PC 端盘符搜索 → 提示不支持
用户："帮我找 D 盘里的工作资料"（目标设备为 PC）

回复："暂不支持按 C 盘/D 盘等具体盘符路径搜索，我帮你用关键词搜索工作资料。"

调用 `SearchLocalFiles`：
```json
{
  "query": "帮我找 D 盘里的工作资料",
  "key_word": ["工作"],
  "time_range": [],
  "location": [],
  "type": [],
  "suffix": [],
  "device": []
}
```

---

### Example 7: 手机端查询文件夹 → 降级处理
用户："找手机里的旅游文件夹"

回复："手机端暂不支持按文件夹搜索，我帮你用关键词搜索相关文件。"

调用 `search_local_files`：
```json
{
  "query": "找手机里的旅游文件夹",
  "key_word": ["旅游"],
  "time_range": [],
  "location": [],
  "type": ["all"],
  "suffix": [],
  "device": []
}
```

## Important Notes

1. **PC 端搜索用 `SearchLocalFiles`，手机端搜索用 `search_local_files`**，不可混用
2. **PC 端上传用 `UploadLocalFiles`，手机端上传用 `upload_local_files`**，不可混用
3. **上传仅在用户明确要求时触发**，搜索完成后不自动上传
4. **上传前必须执行过滤规则**：单文件 >500MB 排除，总大小 >1GB 或数量 >20 截断，并告知用户
5. **`uri` 字段可选**：搜索结果中有则传入，没有只传 `path`
6. **多设备时必须让用户选择**，单设备或用户已明确指定时直接定位
7. **手机端 `key_word` 严格过滤**：不含时间词、类型词、设备词
8. **`folder` 类型仅 PC 端支持**，手机端需提示并降级处理
9. **盘符限制仅适用于 PC 端**，手机端无需此检查
10. **模糊查询必须追问**，不能猜测
11. **时间使用用户本地时区**
12. **搜索返回格式保持 MCP 原样**，由上层展示卡片
